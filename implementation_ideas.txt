PROJECT OVERVIEW
================
An intelligent CI/CD companion that analyzes GitHub Pull Requests against a
Business Requirement Document (BRD), generates summaries for admin approval,
detects workflow changes, and maintains versioned changelogs — all integrated
with Jenkins pipelines.

CORE REQUIREMENTS (From Manager)
=================================
1. Extension/tool that runs alongside GitHub PR approval workflow
2. When a PR merge request is raised against main branch:
   - AI reads and analyzes the merge request contents
   - Generates a summarized view for the admin in a browser extension
   - Admin reviews summary before approving/rejecting the PR
3. On merge approval + completion:
   - Creates a changelog.txt with versioning
   - Includes lines changed per commit
   - Pushes changelog to a SEPARATE repository
4. Workflow change detection:
   - AI maintains persistent memory of the initial BRD workflow
   - Compares every merge against the original BRD
   - Notifies admin if merge will cause workflow changes
   - Shows before/after workflow state
5. Changelog rules:
   - Only technical changes and workflow diagram changes
   - No feature descriptions — pure technical log
6. Must work with Jenkins CI/CD pipelines
7. AI must NOT hallucinate — strong guardrails required


TECH STACK DECISIONS
====================
- AI Model:      NVIDIA NIM, Google Gemini, Groq, OpenRouter, local Ollama (Configurable via env vars)
- Guardrails:    NVIDIA NeMo Guardrails (input/output safety rails)
- Backend:       Python 3.12+ (FastAPI)
- Database:      SQLite / PostgreSQL + AsyncPG (Flexible persistence options)
- Extension:     Chrome Extension (Manifest V3)
- CI/CD:         Jenkins (Generic Webhook Trigger Plugin)
- GitHub API:    PyGithub library + REST API
- Auth:          HMAC-SHA256 webhooks, API key auth

--- COMPONENT 2: AI ENGINE (NVIDIA NIM + NeMo Guardrails) ---

Model: meta/llama-3.1-70b-instruct via NVIDIA NIM API
SDK:   OpenAI Python SDK (NIM is OpenAI-compatible)

Integration pattern:
  from openai import OpenAI

  client = OpenAI(
      base_url="https://integrate.api.nvidia.com/v1",
      api_key=NVIDIA_NIM_API_KEY
  )

  response = client.chat.completions.create(
      model="meta/llama-3.1-70b-instruct",
      messages=[
          {"role": "system", "content": system_prompt_with_brd},
          {"role": "user", "content": f"Analyze this PR diff:\n\n{diff}"}
      ],
      temperature=0.1,
      max_tokens=4096,
      response_format={"type": "json_object"}
  )

Anti-Hallucination Architecture (8 Layers):
  Layer 1: INPUT GROUNDING
    - Only feed actual diff + BRD to the model
    - No internet search, no external context, no training data reliance

  Layer 2: STRUCTURED OUTPUT
    - JSON mode + Pydantic model parsing
    - Every output field is typed and constrained
    - Fields: file, line_range, change_type, description, confidence

  Layer 3: NEMO INPUT RAIL
    - Prompt injection detection on PR content
    - Sanitize code content before sending to AI
    - Block attempts to override system instructions

  Layer 4: NEMO OUTPUT RAIL
    - Fact-checking rail validates AI output
    - Cross-check every file/line reference against actual diff
    - Reject any reference to files not present in the PR

  Layer 5: TEMPERATURE CONTROL
    - temperature=0.1 for near-deterministic responses
    - Minimizes creative/speculative outputs

  Layer 6: CONFIDENCE SCORING
    - AI self-reports confidence (0.0-1.0) for each finding
    - Items below 0.7 flagged for human review
    - Admin sees confidence scores in extension

  Layer 7: DIFF-ONLY PROMPTING
    - System instruction: "Only describe changes present in the diff.
      Do not infer, speculate, or add context beyond the diff and BRD."
    - Explicit anti-hallucination instructions in system prompt

  Layer 8: AUDIT LOGGING
    - Every AI call logged with timestamp, input hash, output, scores
    - Full traceability for debugging and accountability

MULTI-AGENT AUTO-REPAIR LOOP (KILO CODE ARCHITECTURE)
=====================================================
- **Architect Mode**: Plans code modifications based on the PR changes and requirements.
- **Coder Mode**: Writes the concrete code fixes requested by the Architect's plan.
- **Debugger Mode**: Statically verifies the Coder's generated code against syntax rules and functional intent.
- **Self-Healing Engine**: Supports up to 2 verification recovery cycles on errors, providing robust, self-healing branch fixes.
- **GitHub Commit Push**: Automatically pushes valid patches back to the PR branch via GitHub API.

NeMo Guardrails config (config/guardrails.yaml):
  models:
    - type: main
      engine: nim
      model: meta/llama-3.1-70b-instruct
  rails:
    input:
      flows:
        - check prompt injection
        - sanitize code content
    output:
      flows:
        - check hallucination
        - validate file references
        - enforce json schema

Features:
  - Detects when admin is on a GitHub PR page (github.com/*/pull/*)
  - Injects floating panel near the merge button area
  - Shows: PR title, AI summary, workflow impact alerts, changelog preview
  - Color-coded severity:
      GREEN  = No workflow change
      YELLOW = Minor workflow change
      RED    = Major workflow change (blocks merge recommendation)
  - Dark/light theme matching GitHub's current theme
  - Caches PR summaries to avoid redundant API calls
  - API key stored securely in chrome.storage.local
  - **Admin Action Suite**:
      - **Save Edit**: Modify and correct generated summaries.
      - **Compare**: View AI difference detection report between original and edited summaries.
      - **Auto-Repair Code**: Trigger the Architect-Coder-Debugger loop to patch the branch.
      - **Approve & Push**: Instant approval and release changelog publishing.
      - **Reject / Remove**: Remove the PR from the pending queue.

Security:
  - Manifest V3 (no eval, CSP enforced)
  - API calls go through service worker (not content script directly)
  - No remote code loading
  - Token never exposed to content scripts


--- COMPONENT 4: JENKINS PIPELINE ---

Files to create:
  jenkins/
  ├── Jenkinsfile                # Pipeline definition
  └── setup.md                   # Jenkins configuration guide

Jenkinsfile structure:
  pipeline {
      agent any
      triggers {
          GenericTrigger(
              genericVariables: [
                  [key: 'PR_NUMBER', value: '$.number'],
                  [key: 'PR_ACTION', value: '$.action'],
                  [key: 'PR_MERGED', value: '$.pull_request.merged'],
                  [key: 'PR_BASE_REF', value: '$.pull_request.base.ref']
              ],
              causeString: 'PR #$PR_NUMBER $PR_ACTION',
              token: 'capsule-pr-trigger',
              regexpFilterText: '$PR_BASE_REF',
              regexpFilterExpression: '^(main|master)$'
          )
      }
      stages {
          stage('Analyze PR') {
              when { expression { env.PR_ACTION in ['opened', 'synchronize'] } }
              steps { /* Call backend API to analyze PR */ }
          }
          stage('Generate Changelog') {
              when { expression { env.PR_ACTION == 'closed' && env.PR_MERGED == 'true' } }
              steps { /* Trigger changelog generation and push */ }
          }
      }
  }

Required Jenkins plugins:
  - Generic Webhook Trigger Plugin
  - HTTP Request Plugin
  - GitHub Branch Source Plugin
  - Credentials Plugin (for secrets)

Jenkins credentials to configure:
  - CAPSULE_API_URL (backend API base URL)
  - CAPSULE_API_KEY (API key for authentication)
  - GITHUB_TOKEN (for GitHub API access)

SECURITY ARCHITECTURE
=====================

  Concern              | Mitigation
  ---------------------|--------------------------------------------------
  API Key Exposure     | All secrets in Jenkins Credentials + .env (never committed)
  Webhook Forgery      | HMAC-SHA256 signature verification on all webhooks
  Prompt Injection     | NeMo input rail + sanitization before AI processing
  Extension Security   | MV3 (no eval, CSP enforced); chrome.storage for keys
  Data Leakage         | BRD/diffs never sent to external logging services
  Rate Limiting        | Per-IP and per-token limits on all API endpoints
  Audit Trail          | Every AI call logged with timestamp, I/O, confidence
  Least Privilege      | GitHub token scoped to minimum permissions only
