# Program Flow & Architecture Overview

This document describes the overall architecture and step‑by‑step program flow of **Capsule** after the recent enterprise-grade upgrades. It details the interaction between GitHub, the CI/CD runners (GitHub Actions & Jenkins), the asynchronous Celery-Redis backend, the Cloudflare Worker edge, and the Chrome Extension dashboard.

## 2. Step-by-Step Program Flow

### Phase A: Pull Request Analysis (Trigger & Safety Gate)
1. **Trigger**: A developer submits a Pull Request to `main`. This triggers either **GitHub Actions** (`capsule-ci.yml`) or **Jenkins** (`Jenkinsfile`).
2. **Pre-Checks**: The CI/CD runner executes unit and integration tests (`pytest`).
3. **Trigger Analysis**: The runner calls `cli_ci_helper.py --action analyze`, which posts a payload containing the PR number and repository to `/webhooks/jenkins` via the Capsule API.
4. **Task Offloading**: In production, the FastAPI webhook endpoint immediately returns a `202 Accepted` status along with a `task_id` and offloads the analysis to the **Celery Worker** queue via **Redis**.
5. **Polling**: The CI runner polls `/webhooks/task/{task_id}` every 5 seconds until it completes.
6. **AI Analysis (Celery worker)**:
   - Fetches PR details, files, and diffs from the GitHub REST API.
   - Loads the active Business Requirements Document (BRD) via `BRDManager`.
   - Runs a Map-Reduce analysis via the `AIEngine` (using Meta Llama 3.3 70B on NVIDIA NIM) to identify files altered, specific technical changes, and compliance against the BRD.
   - Persists the analysis to the database (saving PR number, repo, branch, summary, changes JSON, workflow impact JSON, and setting `approved = False`).
7. **Compliance and Gate Assessment**:
   - If the AI detects **MAJOR** workflow changes (violating or altering core business flows), and the PR has **not** been approved yet, the polling script prints a critical warning and exits with code `1`.
   - This **fails the CI build** and **blocks the PR from being merged**.
   - If there are minor or no workflow changes, the CI check passes successfully (`exit 0`).
8. **PR Comment Reporting**: The helper script uses the `GITHUB_TOKEN` to automatically post a markdown-formatted analysis summary directly onto the GitHub PR conversation page.

### Phase B: Admin Review & Manual Approval
1. **Notification**: The Chrome Extension detects the pull request page, queries the API, and receives the PR status.
2. **Visual Alert**: The extension badge highlights red (for MAJOR workflow changes) or yellow (for MINOR changes) based on the severity of the impact.
3. **Inspection**: The administrator opens the Chrome Extension popup and reviews the "PR Summary" and the generated "Workflow Diagram".
4. **Admin Actions**:
   - **Save Edit**: The admin can manually correct/edit the generated summary in the extension dashboard. Saving it posts to `/api/pr/{pr_number}/repair` which updates the database.
   - **Compare**: The admin can click "Compare" to trigger `/api/pr/{pr_number}/compare`, which uses OpenRouter to perform an AI difference comparison between the original summary and their edits, recommending actions.
   - **Auto-Repair Code**: If issues are found, the admin can trigger `/api/pr/{pr_number}/auto-repair`. This initiates the Capsule multi-agent loop (Architect-Coder-Debugger) to write, statically verify, and push a code fix directly to the PR branch via GitHub API, resolving bugs automatically.
   - **Reject / Remove**: The admin can reject the PR, removing it from the pending list and database (`POST /api/pr/{pr_number}/reject`).
5. **Approval**: If satisfied, the admin clicks **Approve & Push** (or calls `POST /api/pr/{pr_number}/approve`). This:
   - Updates `approved = True` in the database.
   - Instantly triggers the changelog generation and pushes release updates to the changelog branch, eliminating the need to wait for post-merge actions.
6. **CI Unblocking**: The admin re-runs the failed GitHub Actions job. The polling helper script queries the API, finds `approved = True`, and successfully passes the build, allowing the PR to be merged.

### Phase C: PR Merge & Automated Release Delivery
1. **Trigger**: The PR is merged into the `main` branch. This triggers the GitHub Actions workflow with the `closed` event.
2. **Post-Merge Sync**: The CI pipeline registers that the PR has already been approved and its release assets (changelog, summaries, diagrams) were successfully pushed during Phase B, completing the release pipeline.

### Phase D: Automated 1-Click Onboarding & Infrastructure Setup
1. **Trigger**: An administrator triggers `POST /api/setup-repository` with the repository path and the public URL of the Capsule server.
2. **Changelog Branch Check**: The backend checks if the `changelog` branch exists in the target repository. If missing, it fetches the repository's default branch (e.g. `main`) and programmatically creates `refs/heads/changelog` from the latest commit SHA.
3. **Webhook Configuration**: The backend posts to the GitHub API `/repos/{owner}/{repo}/hooks` to automatically register a webhook configured for `pull_request` events, linking them back to the Capsule `/webhooks/github` endpoint.

## 3. Input & State Variations

| Input Scenario | Flow Trigger | Analysis Output | Gating Action | Changelog Action |
| :--- | :--- | :--- | :--- | :--- |
| **Pure Refactoring / Bugfix**<br>(No BRD impact) | PR Open / Update | Severity: `NONE`<br>Confidence: High | **Passes Instantly**<br>(Checks green, exit 0) | Version: `+0.0.1` (PATCH)<br>Pushed on merge. |
| **New Optional Feature**<br>(Minor BRD impact) | PR Open / Update | Severity: `MINOR`<br>Confidence: High | **Passes Instantly**<br>(Yellow badge alert, exit 0) | Version: `+0.1.0` (MINOR)<br>Pushed on merge. |
| **Workflow Modification**<br>(Major BRD impact, unapproved) | PR Open / Update | Severity: `MAJOR`<br>Confidence: High | **BLOCKED**<br>(Fails build, exit 1, Red badge) | **None**<br>(Merge blocked by GitHub checks) |
| **Workflow Modification**<br>(Major BRD impact, approved) | GHA Re-run after Admin Approval | Severity: `MAJOR`<br>Approved: `True` | **Passes**<br>(Checks green, exit 0) | Version: `+1.0.0` (MAJOR)<br>Pushes changelog & diagram. |

## 4. Component Definitions

- **FastAPI Web App (`backend/`)**: Exposes REST interfaces for GitHub/Jenkins webhook ingestion, automated 1-click repository onboarding (`/api/setup-repository`), task polling, admin approval, weekly consolidated changes, and BRD version uploads.
- **Celery & Redis (`backend/tasks.py`)**: Distributed task queue handling heavy prompt compilation, NVIDIA NIM LLM inference, GitHub commits/files processing, and git push operations.
- **Cloudflare Worker Edge (`cloudflare-worker/`)**: Exposes a proxy gateway for the extension. Uses Workers AI (`stable-diffusion-xl`) for generating workflow diagrams at the edge, bypassing backend server load.
- **Chrome Extension (`extension/`)**: Injects a dashboard widget into the PR page. Displays tabs for:
  - **PR Summary**: File-by-file changes and compliance warnings.
  - **Weekly Changes**: Consolidates approved changes made in the past 1 week.
  - **Workflow Diagram**: Visual system flowchart generated by Workers AI.
